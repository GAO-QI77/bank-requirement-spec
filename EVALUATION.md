# 测评报告 — bank-requirement-spec

评测日期：2026-06-17

## 一、总体评价

这是一份**质量很高、内部高度自洽**的领域 skill。方法论设计成熟：

| 维度 | 评价 |
|---|---|
| 命名 / 结构 | `name` 合法；`.skill` 包内部结构正确（`bank-requirement-spec/SKILL.md` + `references/`），按需加载分层得当 |
| `description` 触发词 | 触发条件丰富（中英双语，覆盖"业务需求转技术需求""第三方代收付""占位符/冻结基线"等多种用户说法），frontmatter 共 740 字符，在 1024 上限内 |
| 工作流可执行性 | 分级路由 A/B/C + 分轮 0/1/2/3A/3B/4，每轮模板自带"基线点验表"和"举证式自检表"，落地性强 |
| 防滥用设计 | 十条硬禁令 / 占位符 / 高敏感概念边界框架 / 反锚定原则，针对弱模型典型失败逐条设防，设计哲学（让"做对"成为最省力路径）一致贯穿 |
| 诚实边界 | SKILL / README / core-engine 三处都明确列出"兜不住"的情形，不夸大 |

结论：**可用，方法论无重大缺陷**。本次改进集中在"交付工程化"与一处真实的占位符不一致。

## 二、评测方法

没有停留在通读，而是把 skill 自身反复建议的"形式核验脚本"实现出来（`tools/consistency_check.py`），对跨文件约束做数据驱动核验：

- **C1** 模板中出现的占位符是否都在 core-engine「固定占位符表」中登记
- **C2** 引用的不涉及原因代码 `N0x` 是否都已定义
- **C3** SKILL「轮次挂载表」提到的第三方章节，third-party.md 是否都有对应小节
- **C4** 六轮轮次命名是否在各文件一致
- **C5** 版本号是否一致

## 三、发现的问题与处置

| 编号 | 问题 | 性质 | 处置 |
|---|---|---|---|
| P1 | 交付包混乱：`files.zip` 同时含打好的 `.skill` 与一批散落重复副本（根目录 `core-engine.md` 等），而 `SKILL.md` 引用的是 `references/core-engine.md`——散落副本路径是断的 | 工程/交付 | 重建为干净仓库结构，源唯一、产物在 `dist/` |
| P2 | **占位符集合未闭合**：`third-party.md` / `human-checklist.md` 使用了 3 个未在固定占位符表登记的变体——`【待业务/财会/法务条线确认】`、`【待数据治理条线确认】`、`【待风险合规条线确认】`。这直接削弱 skill 的核心纪律"仅允许原样输出占位符"，并会让 checklist 推荐的"扫描未登记占位符"脚本误报合法输出 | **内容缺陷** | 在 core-engine 固定占位符表中登记这 3 个变体，闭合集合（C1 现已通过） |
| P3 | 缺 skill 自身反复建议的"形式核验脚本"（残余风险表、一票否决项均提"建议写成脚本/半天可写完"），但原始交付未提供 | 缺失 | 新增 `tools/consistency_check.py`（C1–C5）与 `tools/build_skill.sh`（校验+打包） |
| P4 | README 声明 MIT 但无 `LICENSE` 文件 | 合规/工程 | 补 `LICENSE`（MIT） |

误报澄清：校验脚本初版有两处自身 bug（markdown 表头分隔行 `---` 截断了表区段、`【】` 同时用于占位符与章节标题），已修正后 C2 与占位符识别恢复正确——这两条不是 skill 的问题。

## 四、改进后状态

```
$ python3 tools/consistency_check.py
  C1 OK  占位符全部登记在册（登记 20 个）
  C2 OK  原因代码引用合法（定义 10 个）
  C3 OK  第三方挂载章节齐全
  C4 OK  六轮模板命名一致
  C5 OK  版本号 V4.0 一致
  全部通过：内部引用一致。
```

`./tools/build_skill.sh` 在校验通过后产出 `dist/bank-requirement-spec.skill`，可直接安装。

## 五、未改动的设计取舍（说明而非缺陷）

- **`description` 兼含"做什么"与"何时用"**：按 Anthropic CSO 建议，description 宜聚焦触发条件而非复述流程。但本 skill 的双语触发词已足够丰富、且对国有大行用户的检索习惯友好，过度精简反而损失可发现性，故保留。
- **弱模型在 C 档长基线下静默丢内容**：属模型能力上限，skill 已诚实标注并给出"实测基线承载上限 / 改结构化摘要传递"的缓解，非文档可消除。

这两点保持原样是有意为之，不作为缺陷计。
