# 银行业务需求转技术需求说明书生成器（V4.0）

## 这个 skill 是什么

这是一套**给行内弱模型用、要真正投产**的需求书生成方法论 + 可直接投喂的分轮 prompt 模板。它的设计哲学是：**让"做对"成为最省力的路径**——通过分级路由、占位符、冻结基线、举证式自检，把"不要编造""不要越权"这类需要模型自律的开放要求，变成模型能可靠执行的封闭动作。

它不是一次性吐出整份文档，而是**分 0/1/2/3A/3B/4 六轮**，每轮之间有人工确认。

## 何时用哪个文件

```
SKILL.md (本文件) ……… 工作流总控、档位判定、何时加载扩展包
references/
├── core-engine.md …… V4.0 核心引擎：模型规则、占位符表、原因代码、
│                       分级路由、业务条线对照表、高敏感概念框架、六轮模板、Few-shot
├── third-party.md …… 第三方合作业务域扩展包（仅第三方合作类需求加载）
└── human-checklist.md  人工确认 Checklist（把关人用，不投喂模型）
```

**加载顺序与判断逻辑：**

1. **任何需求**：先读 `references/core-engine.md`。这是骨架，所有需求都走它。
2. **第三方合作类需求**：在第 0 轮判定档位时，若命中第三方合作信号（见下），**额外加载** `references/third-party.md`，把它的业务域章节挂到 core-engine 的对应轮次上。
3. **做人工把关**：把关人/评审者读 `references/human-checklist.md`，它不投喂模型。

## 核心工作流（必须按此执行）

### 第一步：判定是不是"第三方合作类需求"

读业务原文，搜以下信号词。命中任意一个，即按第三方合作类处理，加载 `third-party.md`：

> 第三方、外部机构、合作机构、代收、代付、代扣、批量扣款、批量入账、代发、归集、签约、协议、对账、回盘、清算、清分、手续费、分润、委托单位、商户、平台方、供应链、水电燃气、社保医保公积金、税务财政、保险公司、基金公司、证券公司、缴费、扣费、退费、数据报送、账户验证、身份核验、数据采集、数据回传

没命中，就是普通单条线需求，只用 core-engine 即可。

### 第二步：判定档位（A / B / C）

档位定义与升档关键词见 `core-engine.md` 第二部分。简述：

- **A 档**：纯页面/字段/文案/查询/性能，无资金无账务无第三方无授权无敏感 → 走 0 / 1+2合并 / 4，三次调用。
- **B 档**：业务规则/参数/接口/渠道流程/状态流转，不直接涉核心账务 → 走 0 / 1 / 2 / 3A或3B / 4。
- **C 档**：资金/账户/核心账务/第三方/反向交易/远程授权/对账清算/风控/监管/敏感信息 → 走完整 0 / 1 / 2 / 3A / 3B / 4，**不得合并**。

⚠️ **第三方合作类需求几乎必然落入 B 或 C 档**（涉及外部系统对接，至少命中强触发词"第三方/接口/对账"）。纯第三方查询展示且不涉资金的，最低也是 B 档。

### 第三步：按轮次生成

每一轮调用时，给模型投喂的内容 = **core-engine 第一部分「模型规则」（每轮必带）** + **本轮模板** + **（第三方合作类时）third-party 对应业务域章节** + **上一轮冻结基线**。

具体每轮模板见 `core-engine.md` 第五部分。第三方合作类需求在以下轮次需挂载 `third-party.md` 的对应内容：

| 轮次 | 普通需求 | 第三方合作类额外挂载 |
|---|---|---|
| 第 0 轮 | 原文质量 + 档位判定 | + 合作机构类型识别、业务类型分类 |
| 第 1 轮 | 需求识别澄清 | + 第三方接入生命周期定位（准入/运营/退出哪一段） |
| 第 2 轮 | 功能 + 规则 | + 数据采集与使用要求章节 |
| 第 3A 轮 | 系统交互 + 账务 | + 第三方系统边界、对账清算、清算资金划转、手续费计收 |
| 第 3B 轮 | 差错/授权/风控/安全/凭证/非功能/测试 | + 第三方差错矩阵、报文安全、第三方数据使用边界、服务中断应急 |
| 第 4 轮 | 整合评审版 | + 第三方接入流程、参数配置、合作退出机制 等章节 |

### 第四步：交付与人工确认

- 终稿（第 4 轮）正文供评审；过程性产物（自检表/占位符清单/疑点）放附录。
- 把关人对照 `references/human-checklist.md` 核验。**没有人工 checklist 兜底，不允许投产。**

## 四条不可违反的底线（贯穿所有轮次）

1. **不编造**：交易码、接口编号、系统名、内部户、会计科目、清算账户、手续费比例、清算周期、制度文号、产品代码——业务原文没给的，一律用 core-engine 的固定占位符，禁止填具体值。
2. **不越权**：不替架构/开发做技术选型（Redis/MQ/缓存/索引/协议……），统一写"待架构/开发评估"；不替财会下账务口径；不替合规下监管判断。
3. **不下高敏感结论**：反向类（抹账/冲正/红冲/撤销/退款）、客户主体状态类（销户/身故/止付/冻结）等，只输出"待确认问题清单"，不输出概念定性。
4. **不糊弄歧义**：待确认问题必须具体到字段/流程/口径，禁止"详情待业务确认"这类空话；功能点不得以待确认问题作为确定依据。

## 这个 skill 不能做什么（诚实边界）

- 业务原文本身写错或写漏，本 skill 兜不住（垃圾进垃圾出）。第 0 轮的原文质量检查只能拦明显缺口。
- 行内会计/合规/风控的最终口径，本 skill 不下结论，必须条线确认。
- 弱模型在 C 档长基线下可能静默丢内容，这是模型能力上限。试点首要任务是实测"基线承载上限"，超载则把基线改为结构化摘要传递。
- 占位符再多也代替不了行内规范库。占位符是"标记不知道"，不是"知道"。
